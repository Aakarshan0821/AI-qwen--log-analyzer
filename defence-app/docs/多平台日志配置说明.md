# Defence App 多日志源兼容说明

## 结论

`defence-app` 现在可以不只接 `pfSense`，也可以接入：

- `OPNsense` 防火墙日志
- `WAF` 日志
- `Nginx / Apache / IIS` Web 访问日志
- `Linux syslog / auth.log / auditd`
- `Windows Event / Sysmon`
- 其他已经写入 `Elasticsearch`，并且至少包含时间、IP、动作、消息等关键字段的日志

这次已在项目中补充了“多来源兼容解析层”：

- 优先识别 `ECS` 标准字段
- 兼容 `pfSense` 的 `pfsense.columnN`
- 兼容 `OPNsense` 的 `opnsense.columnN`
- 兼容部分常见 `WAF / Web / 服务器日志` 的别名字段

因此，项目已经从“pfSense 专用”升级为“基于 Elasticsearch 的通用安全/访问日志分析前端”。  
如果你的日志已经被 `Logstash / Filebeat / Elastic Agent` 规范化到 ECS，通常只需要少量接入配置即可投入使用。

## 当前版本适合什么场景

适合直接使用：

- 防火墙日志已经进入 ES，且存在 `source.ip`、`destination.ip`、`event.action`
- WAF 日志已经进入 ES，且能拿到客户端 IP、服务器 IP、动作和消息
- Nginx / Apache / IIS 访问日志已经用 `Filebeat` 或 `Elastic Agent` 写入 ES
- Linux / Windows 日志已经进入 ES，并带有 ECS 或相近字段

需要再补一层解析后再用：

- 原始文本日志直接入 ES，没有拆字段
- 日志只有 `message`，没有 `source/destination/client/server` 等结构化字段
- 某些厂商 WAF 日志字段非常私有，既不是 ECS，也没有统一别名

不建议直接投入生产的场景：

- ES 中日志完全未清洗，只有大段原始字符串
- 需要精确攻击规则、会话上下文、资产关联、用户画像、威胁情报联动
- 需要多租户、权限分级、审计追踪、报表导出、告警通知编排

## 推荐接入方式

### 1. 最推荐：统一映射到 ECS

建议你的日志在进入 Elasticsearch 前，尽量映射为 ECS：

- 时间：`@timestamp`
- 来源：`source.ip` / `source.port`
- 目标：`destination.ip` / `destination.port`
- 客户端：`client.ip` / `client.port`
- 服务端：`server.ip` / `server.port`
- 动作：`event.action`
- 原始消息：`message`
- 原始全文：`event.original` 或 `log.original`

这样 `defence-app` 基本可以直接使用，后续扩展规则也最省事。

### 2. 防火墙日志

适合：

- `pfSense`
- `OPNsense`
- 其他能拆成源/目标 IP 和端口的防火墙日志

建议：

- 用 `Logstash` 或 `Elastic Agent` 拆字段
- 最终保留 `source.*`、`destination.*`、`event.action`

### 3. WAF 日志

主流可接入来源：

- `ModSecurity`
- `NGINX App Protect WAF`
- `F5` WAF
- `FortiWeb`
- `Cloud WAF` 导出的结构化安全日志

建议：

- 至少把客户端 IP 映射到 `client.ip` 或 `source.ip`
- 把受保护服务 IP 映射到 `server.ip` 或 `destination.ip`
- 把拦截/放行结果映射到 `event.action` 或 `rule.action`
- 把规则名、攻击摘要映射到 `message` / `rule.name`

### 4. Web 服务器日志

主流可接入来源：

- `Nginx access`
- `Nginx error`
- `Apache access`
- `Apache error`
- `IIS access`

建议：

- 通过 `Filebeat nginx/apache/iis module` 或 `Elastic Agent` 采集
- 客户端 IP 通常进入 `source.ip` 或 `client.ip`
- 服务端 IP/端口进入 `server.ip` / `server.port`
- 访问结果可映射到 `event.action`、`http.response.status_code`

说明：

- 这类日志更适合做访问画像、异常来源识别、暴力扫描发现
- 如果没有明确的拦截动作，`block` 规则不一定触发，但 IP 频率、异常访问等仍可分析

### 5. Linux 服务器日志

主流可接入来源：

- `syslog`
- `auth.log`
- `secure`
- `auditd`
- `journal`

建议：

- 使用 `Elastic Agent` 或 `Filebeat system module`
- 如果日志带有远端访问方，尽量提取到 `source.ip`
- 登录失败、拒绝、封禁等结果可映射到 `event.action`

### 6. Windows 服务器日志

主流可接入来源：

- `Security`
- `System`
- `Application`
- `Sysmon`
- `PowerShell`

建议：

- 使用 `Winlogbeat` 或 `Elastic Agent`
- 如果是登录、网络、RDP、进程网络事件，尽量补出 `source.ip` / `destination.ip`
- 行为动作统一落到 `event.action`

## 目前对投入使用的建议

### 可以较快投入使用

如果你满足下面条件，可以较快上线试用：

- Elasticsearch 已稳定运行
- 日志已经能进入 ES
- 至少一部分日志具有结构化 IP、端口、动作字段
- 你接受当前产品定位是“轻量检索 + AI 辅助分析 + 基础告警”

### 上线前建议做的最小改造

1. 把各类日志尽量统一到 ECS 字段
2. 将不同设备分到清晰索引，例如：
   - `firewall-*`
   - `waf-*`
   - `nginx-*`
   - `linux-*`
   - `windows-*`
3. 在 `.env` 中把 `ES_INDEX_PATTERN` 设成你的联合模式，例如：

```env
ES_INDEX_PATTERN=firewall-*,waf-*,nginx-*,linux-*,windows-*
```

4. 为高危端口、自动分析排除规则做一次初始化
5. 用 1 到 3 天的真实日志做误报验证

## 已知边界

当前检测规则仍然偏“网络安全日志”风格，最适合：

- 防火墙
- WAF
- 反向代理访问日志
- 登录/连接类服务器日志

对于以下场景，建议后续继续增强：

- Web 攻击类型分类
- HTTP 状态码与 URL 路径级异常规则
- 用户维度登录失败聚合
- Windows 事件 ID 专项规则
- Linux SSH 爆破专项规则
- 告警通知与工单联动

## 推荐的主流日志清单

你可以优先接入下面这些“最有价值且最常见”的日志：

- `pfSense / OPNsense` 防火墙日志
- `ModSecurity / NGINX App Protect / FortiWeb` WAF 日志
- `Nginx / Apache / IIS` 访问日志
- `Linux auth.log / secure / auditd / syslog`
- `Windows Security / Sysmon / PowerShell`
- `VPN / RDP / SSH` 访问相关日志
- `Suricata / Zeek` 等网络安全检测日志

## 对你当前项目的建议

如果你现在的目标是“尽快可用”，建议分 3 步：

1. 先接 `OPNsense / WAF / Nginx / Linux auth` 这几类最常见日志
2. 在 Logstash 或 Agent 层统一映射到 ECS
3. 先让 `defence-app` 承担统一查询、轻量告警和 AI 摘要，再逐步补更细规则

这样是可行的，而且改动量相对可控。
