# Defence App 使用与部署说明

## 1. 文档目的

本文档用于说明 `Defence App` 的本地使用方法、运行环境要求、配置项含义、单机部署方式、反向代理部署方式，以及常见问题排查方法。

如果你需要完整的 `ELK Docker 部署`、`pfSense 远程日志接入`、`column 字段拆分`、`开机自启动` 和 `QWEN_API_KEY` 配置位置说明，请同时参考 `docs/elk-docker-pfsense-deployment.md`。

如果你需要接入 `WAF / OPNsense / Nginx / Apache / Linux / Windows` 等多日志源，请同时参考 `docs/multi-source-log-compatibility.md`。

如果你需要登录模块的详细说明（含环境变量、认证行为、页面交互与排错），请参考 `docs/login-module-guide.md`。

适用场景：

- 本地开发和调试
- 实验室或内网服务器部署
- Ubuntu 单机长期运行
- Windows 环境快速启动验证

## 2. 项目简介

`Defence App` 是一个基于 `FastAPI + Jinja2 + 原生 JavaScript` 的轻量日志分析平台，主要用于：

1. 按 IP 或时间范围检索 pfSense 日志
2. 定时分析最近一段时间的可疑事件
3. 手动触发最近窗口分析与自定义时间分析
4. 对勾选日志直接发起 AI 总结
5. 使用千问兼容接口生成分析摘要
6. 通过网页查看日志、检测结果和系统健康状态

## 3. 功能概览

当前版本支持以下能力：

- 日志检索
  - 按 IP 检索
  - 按时间范围检索
  - 限制返回数量
  - 勾选日志、全选当前结果、清空已选
- 可疑事件检测
  - 识别 `block` 动作
  - 识别访问高危目标端口的事件
  - 识别时间窗口内高频出现的来源 IP
  - 对同一事件的多条规则命中做归并
- AI 总结
  - 左侧菜单切换“查找日志”和“AI分析”
  - 每 10 分钟自动刷新最近窗口分析结果
  - 支持自定义时间范围直接生成 AI 总结
  - 支持对勾选日志直接生成 AI 总结
  - 支持发送测试消息检查 AI 连通性
  - 配置 `QWEN_API_KEY` 后调用千问兼容接口生成简要分析
  - 未配置 AI 时自动使用本地摘要
- 运行状态检查
  - 提供 `/api/health` 健康检查接口
  - 前端显示 Elasticsearch、AI 和定时任务状态

## 5. 软件环境要求

- Python `3.10` 或更高版本
- 可访问的 Elasticsearch 服务
- 已写入 pfSense 日志的索引，默认模式为 `pfsense-*`

### 5.2 推荐运行环境

- 开发环境：Windows 10/11 或 Ubuntu 22.04+
- 部署环境：Ubuntu 22.04 / 24.04
- Python 虚拟环境：`venv`
- Web 服务启动方式：`uvicorn`

- `apscheduler`

pip install -r requirements.txt

### 6.1 配置项详解

#### Elasticsearch 相关

- `ES_URL`
  - Elasticsearch 地址
  - 示例：`http://127.0.0.1:9200`

- `ES_INDEX_PATTERN`
  - 日志索引模式
  - 示例：`pfsense-*`

- `ES_USERNAME`
  - Elasticsearch 用户名
  - 未启用认证时可留空

- `ES_PASSWORD`
  - Elasticsearch 密码
  - 未启用认证时可留空

- `ES_TIMEOUT_SECONDS`
  - Elasticsearch 请求超时秒数
  - 建议：`10` 到 `30`

- `ES_SEARCH_BATCH_SIZE`
  - 检测任务分批读取日志时的每批条数
  - 建议：`200` 到 `1000`

- `DETECTION_MAX_LOGS`
  - 单次检测最多分析的日志数
  - 超出该数量时会截断并返回 `logs_truncated=true`

#### AI 相关

- `QWEN_API_KEY`
  - 千问兼容接口密钥
  - 不填写时，系统只输出本地规则摘要

- `QWEN_BASE_URL`
  - 千问兼容接口地址

- `QWEN_MODEL`
  - 使用的模型名称
  - 示例：`qwen-plus`

- `AI_TIMEOUT_SECONDS`
  - AI 请求超时秒数

#### 检测逻辑相关

- `DETECTION_WINDOW_MINUTES`
  - 定时检测的窗口大小，单位为分钟

- `SUSPICIOUS_THRESHOLD`
  - 单个来源 IP 在检测窗口内出现多少次后会被认定为高频来源 IP

- `RISKY_PORTS`
  - 需要重点关注的高危端口
  - 逗号分隔，例如：`22,23,3389,445,1433,3306`

- `MAX_ALERTS_DISPLAY`
  - 前端最多展示多少条告警

#### 运行控制相关

- `ENABLE_SCHEDULER`
  - 是否启用内置定时检测任务
  - 单实例部署可设为 `true`
  - 多实例部署建议设为 `false`

- `AUTO_REFRESH_SECONDS`
  - 前端自动刷新最新检测结果和健康状态的间隔秒数

#### 登录认证相关

- `AUTH_ENABLED`
  - 是否启用登录认证
  - 默认：`true`

- `LOGIN_EMAIL`
  - 登录账号（邮箱）

- `LOGIN_PASSWORD`
  - 登录密码

- `SESSION_SECRET`
  - 会话签名密钥
  - 生产环境必须替换为高强度随机值

- `SESSION_MAX_AGE_SECONDS`
  - 默认会话有效期（秒）
  - 勾选“记住 30 天”后将按 30 倍时长生效

### 7.3 页面使用方法

#### 左侧菜单

首页左侧提供两个菜单入口：

1. `查找日志`
2. `AI分析`

切换菜单不会刷新页面，已勾选日志会保留在当前会话中。

#### 查找日志

1. 打开首页
2. 点击左侧 `查找日志`
3. 在查询区域填写以下条件：
   - IP 地址
   - 开始时间
   - 结束时间
   - 返回条数
4. 点击“查询日志”
5. 页面会以卡片形式显示匹配日志
6. 如需送入 AI，可逐条勾选日志，或点击“全选当前结果”
7. 点击“清空已选”可清除当前勾选状态
8. 点击“分析已选日志”后，会自动切换到 `AI分析` 视图展示结果

#### AI分析

`AI分析` 视图包含最近窗口自动分析、自定义时间分析、已选日志分析和 AI 连接测试 4 个区域。

#### 最近窗口自动分析

1. 点击左侧 `AI分析`
2. 在“最近窗口自动分析”区域查看最近一次分析结果
3. 点击“刷新最新结果”可获取最近一次自动或手动分析结果
4. 点击“立即分析最近窗口”可按输入分钟数重新执行一次
5. 页面会展示：
   - 最近分析时间
   - 检查日志数
   - 可疑事件总数
   - AI 总结
   - 告警卡片列表

#### 自定义时间分析

1. 在“自定义时间分析”区域填写开始时间、结束时间和分析条数上限
2. 点击“开始自定义时间分析”
3. 系统会拉取该时间范围内日志，并直接生成 AI 总结
4. 页面会展示时间范围、分析日志数和 AI 总结

#### 已选日志分析

1. 先在 `查找日志` 视图勾选一个或多个日志
2. 切换到 `AI分析`
3. 点击“分析已选日志”
4. 页面会展示已选日志数量和 AI 总结

#### AI 连接测试

1. 在“AI 连接状态”区域输入测试消息
2. 点击“测试 AI 连接”
3. 页面会展示 AI 状态、测试消息和 AI 返回内容

## 8. 接口说明

### 8.1 `GET /`

主页页面。

### 8.2 `GET /api/search`

按 IP 或时间范围检索日志。

常用参数：

- `ip`：可选，IP 地址
- `start`：可选，开始时间，ISO8601 格式
- `end`：可选，结束时间，ISO8601 格式
- `size`：返回条数，默认 `200`

示例：

```bash
curl "http://127.0.0.1:7860/api/search?ip=192.168.1.1&size=20"
```

返回日志中常见字段包括：

- `id`
- `index`
- `timestamp`
- `event_action`
- `source_ip`
- `destination_ip`
- `destination_port`
- `message`

### 8.3 `POST /api/detect/manual`

手动触发最近窗口检测。

参数：

- `minutes`：检测窗口，范围 `1` 到 `1440`

示例：

```bash
curl -X POST "http://127.0.0.1:7860/api/detect/manual?minutes=10"
```

### 8.4 `GET /api/detect/latest`

获取最近一次检测结果。

示例：

```bash
curl "http://127.0.0.1:7860/api/detect/latest"
```

### 8.5 `GET /api/ai/status`

获取当前 AI 配置状态。

示例：

```bash
curl "http://127.0.0.1:7860/api/ai/status"
```

### 8.6 `POST /api/ai/test`

发送一条测试消息到 AI。

示例：

```bash
curl -X POST "http://127.0.0.1:7860/api/ai/test?message=请回复AI连接正常"
```

### 8.7 `POST /api/ai/analyze/range`

按自定义时间范围拉取日志并直接生成 AI 总结。

示例：

```bash
curl -X POST "http://127.0.0.1:7860/api/ai/analyze/range" -H "Content-Type: application/json" -d "{\"start\":\"2026-03-14T11:00:00Z\",\"end\":\"2026-03-14T12:00:00Z\",\"size\":200}"
```

### 8.8 `POST /api/ai/analyze/selected`

对前端勾选的日志直接生成 AI 总结。

示例：

```bash
curl -X POST "http://127.0.0.1:7860/api/ai/analyze/selected" -H "Content-Type: application/json" -d "{\"selected_logs\":[{\"timestamp\":\"2026-03-14T12:00:00Z\",\"event_action\":\"block\",\"source_ip\":\"1.1.1.1\",\"destination_ip\":\"2.2.2.2\",\"destination_port\":3389,\"message\":\"blocked traffic\"}]}"
```

### 8.9 `GET /api/health`

查看服务健康状态。

示例：

```bash
curl "http://127.0.0.1:7860/api/health"
```

返回内容包含：

- 应用状态
- Elasticsearch 状态
- AI 状态
- 定时任务状态

## 9. 检测逻辑说明

当前默认规则如下：

1. 若日志事件的 `event.action == block`，则记为阻断流量
2. 若目标端口在 `RISKY_PORTS` 中，则记为高危目标端口
3. 若单个来源 IP 在检测窗口内出现次数大于等于 `SUSPICIOUS_THRESHOLD`，则生成高频来源 IP 聚合告警
4. 若同一日志同时命中多条规则，只保留一条告警，并记录多个命中原因
5. 若检测日志数量达到 `DETECTION_MAX_LOGS` 上限，系统会标记结果被截断

## 11. 使用 systemd 守护运行

推荐在 Ubuntu 上使用 `systemd` 管理服务。

### 11.1 创建服务文件

新建文件 `/etc/systemd/system/defence-app.service`：

```ini
[Unit]
Description=Defence App
After=network.target

[Service]
Type=simple
User=user
WorkingDirectory=/home/user/defence-app
EnvironmentFile=/home/user/defence-app/.env
ExecStart=/home/user/defence-app/.venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 7860
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

注意：

- 请把 `User=ubuntu` 改成你的实际运行用户
- 请确认 `WorkingDirectory` 和 `ExecStart` 路径正确

### 11.2 启动服务

```bash
sudo systemctl daemon-reload
sudo systemctl enable defence-app
sudo systemctl start defence-app
```

### 11.3 查看运行状态

```bash
sudo systemctl status defence-app
```

查看日志：

```bash
journalctl -u defence-app -f
```

## 12. 使用 Nginx 反向代理

如果你希望通过 `80` 或 `443` 端口提供访问，建议在前面加一层 `Nginx`。

### 12.1 安装 Nginx

```bash
sudo apt update
sudo apt install -y nginx
```

### 12.2 新建站点配置

新建 `/etc/nginx/sites-available/defence-app`：

```nginx
server {
    listen 80;
    server_name _;

    location / {
        proxy_pass http://127.0.0.1:7860;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

启用配置：

```bash
sudo ln -s /etc/nginx/sites-available/defence-app /etc/nginx/sites-enabled/defence-app
sudo nginx -t
sudo systemctl reload nginx
```

### 12.3 HTTPS 建议

如果系统面向正式网络环境，建议再用 `certbot` 配置 HTTPS。

## 13. 生产部署建议

### 13.1 关于定时任务

当前项目内置 APScheduler 定时检测任务。

如果你采用以下部署模式，建议关闭内置调度器：

- 多实例部署
- 多个 `uvicorn` worker
- 由外部调度平台触发检测

此时建议：

```env
ENABLE_SCHEDULER=false
```

然后由外部方式调用：

- `POST /api/detect/manual`
- `POST /api/ai/analyze/range`
- 或单独实现脚本/定时任务

### 13.2 关于日志量

如果 pfSense 日志量较大，建议重点调整：

- `ES_SEARCH_BATCH_SIZE`
- `DETECTION_MAX_LOGS`
- `DETECTION_WINDOW_MINUTES`

避免一次检测拉取过大的窗口数据。

### 13.3 关于安全

当前项目已内置登录认证。如果部署在共享网络中，建议至少做到：

- 使用高强度 `LOGIN_PASSWORD` 和 `SESSION_SECRET`
- 只开放内网访问
- 通过 Nginx 或网关加访问控制
- 使用 HTTPS
- 不要把 `.env` 提交到代码仓库

## 15. 常见问题排查

### 15.1 页面打不开

排查项：

- `uvicorn` 是否启动成功
- 7860 端口是否被防火墙拦截
- Nginx 反向代理是否配置正确

### 15.2 `/api/health` 显示 Elasticsearch 为 down

排查项：

- `ES_URL` 是否可访问
- 索引是否存在
- ES 是否要求认证
- `ES_USERNAME` 和 `ES_PASSWORD` 是否正确

### 15.3 日志检索无结果

排查项：

- `ES_INDEX_PATTERN` 是否正确
- 时间范围是否合理
- IP 是否写错
- Elasticsearch 中是否确实存在 pfSense 日志

### 15.4 最近窗口分析或自定义时间分析失败

排查项：

- Elasticsearch 是否可用
- 检测时间窗口是否过大
- `DETECTION_MAX_LOGS` 是否过小
- 后端日志中是否存在超时或解析异常

### 15.5 已选日志分析没有结果

排查项：

- 是否已经在 `查找日志` 里勾选日志
- 勾选后是否误点了“清空已选”
- 查询结果中的日志是否确实已展示

### 15.6 AI 总结未返回

排查项：

- `QWEN_API_KEY` 是否填写
- `QWEN_BASE_URL` 是否正确
- 外网是否可访问 AI 接口
- `AI_TIMEOUT_SECONDS` 是否过小

如果 AI 不可用，系统仍会返回本地规则摘要，这属于正常降级行为。

建议维护时优先关注以下文件：

- `app/services/detection_service.py`
